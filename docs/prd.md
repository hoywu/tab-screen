# Tab Screen PRD

**文档信息**

- 项目代号: `tab-screen`
- 文档类型: Product Requirements Document
- 文档目标: 为后续 Rust 服务端与 Flutter Android 客户端实现提供统一、可执行的产品与工程约束
- 当前版本: `v1.0`
- 文档状态: Draft for implementation

**产品概述**

`tab-screen` 是一个将 Android 平板扩展为 Linux 桌面副屏的本地优先产品。

服务端运行在 Linux Wayland 桌面环境中，负责在客户端成功连接后按需创建虚拟显示输出、捕获该输出的画面、按照可配置策略编码并推送到客户端。客户端运行在 Android 平板上，负责建立连接、上报自身设备屏幕参数、协商串流参数、硬件解码并以现代化全屏界面展示桌面画面。

产品必须同时支持两种连接方式。

- USB 连接: 首版通过 `adb reverse` 或等价 ADB 端口转发实现。
- 局域网连接: 客户端手动输入服务端 `IP:port` 连接。

产品必须支持双向参数协商。

- 服务端定义允许范围、默认值、偏好值和硬性限制。
- 客户端必须在握手阶段上报自身设备屏幕参数，至少包括物理分辨率、刷新率、色深、方向和 DPI 或等效逻辑密度信息。
- 虚拟屏幕的默认参数必须以客户端上报的设备屏幕参数为基础生成，除非服务端配置中存在显式强制覆盖。
- 客户端默认遵循服务端偏好，也可以在设置页请求覆盖部分串流参数。
- 服务端对客户端请求进行能力校验、约束裁剪和最终确认，并将实际生效结果返回客户端。

**问题陈述**

现有跨设备副屏方案通常存在以下问题。

- Linux Wayland 支持不完整，尤其是虚拟显示输出能力存在桌面环境差异。
- 开源方案常将重点放在远程桌面，而不是本地副屏场景。
- 可配置性不足，无法精细控制虚拟屏幕参数、编码参数、硬件编码策略和运行方式。
- Android 客户端 UI 和连接体验往往偏工程化，不适合作为日常使用应用。
- 很少有方案将“设备接入后自动出现稳定可识别的新显示器”作为一等能力，导致桌面环境难以记住每台平板的布局与缩放配置。

`tab-screen` 的目标是提供一个本地优先、配置清晰、便于 systemd 管理、便于后续迭代的副屏系统。

**产品目标**

- 将 Android 平板作为 Linux 桌面的扩展显示器使用，而不是简单镜像或远程控制窗口。
- 服务端以 Rust 实现，提供稳定的 CLI、配置文件和 systemd 集成能力。
- 客户端以 Flutter 实现，提供现代化 UI、沉浸式显示和设置页。
- 首版支持通过 USB 和局域网建立连接。
- 用户可分别配置虚拟屏幕参数和串流参数。
- 默认情况下，虚拟屏幕参数应跟随客户端实际上报的屏幕能力，尽量实现“像插入一块新显示器一样”的行为。
- 客户端可请求覆盖串流参数，服务端负责协商并动态应用。
- 服务端与客户端都提供清晰的诊断信息，便于排查编码、连接和兼容性问题。
- 在服务端作为 `systemd --user` 开机自启动服务运行时，客户端打开后应可自动尝试连接上次成功的服务端地址，并在连接成功后让 Linux 桌面自动出现新显示器。

**非目标**

- 首版不追求对所有 Wayland compositor 的通用虚拟显示兼容。
- 首版不实现跨公网访问。
- 首版不实现音频串流。
- 首版不实现触控反向输入、键鼠共享、剪贴板同步、文件传输。
- 首版不实现多客户端同时观看同一画面。
- 首版不实现多块 Android 平板同时作为多个副屏。
- 首版不承诺 HDR、10-bit、AV1、色彩管理全链路支持。

**目标用户**

- 使用 Linux Wayland 桌面的开发者、创作者和效率工具重度用户。
- 拥有 Android 平板，希望其作为低成本扩展显示器的用户。
- 能接受基础配置和开发者工具操作的早期用户。

**核心使用场景**

- 用户在 Linux 笔记本或台式机上启动服务端，将 Android 平板作为额外显示区域放置聊天工具、监控面板、文档或参考资料。
- 用户通过 USB 连接平板，在不依赖 Wi-Fi 稳定性的情况下使用副屏。
- 用户在局域网环境中输入服务端地址快速连接平板作为副屏。
- 用户在 Android 客户端选择“遵循服务器偏好”或手动指定 `1080p / 60fps / H.264 / hardware decode` 等偏好。
- 用户将服务端设为 `systemd --user` 开机自启动后，无需先手动操作 Linux 端；只要打开平板客户端并成功连接上次使用的服务端地址，Linux 桌面就会像连接了新显示器一样自动出现一个新的扩展显示器。
- 服务端根据机器能力和配置限制，自动决定实际编码参数并将结果同步给客户端。

**关键产品原则**

- 本地优先: 默认服务于同一物理环境内的设备连接。
- 配置优先: 所有关键行为都应可通过配置文件声明，而不是硬编码。
- 明确协商: 客户端请求不等于最终生效，所有实际值必须有确认结果。
- 接入即显示: 虚拟显示器必须按客户端连接生命周期创建和销毁，而不是长期常驻。
- 故障可诊断: 每一类失败都应具备明确错误码、日志与用户可理解提示。
- 渐进兼容: 明确区分 MVP 支持范围与后续扩展范围，避免在 v1 过度承诺。

**约束与关键决策**

- Wayland 下“创建虚拟显示器”并非完全标准化能力，不同 compositor 支持方式不同，因此 v1 必须限制支持范围并为显示后端做抽象。
- 服务端应优先以 `systemd --user` 方式运行，而不是系统级 root service，因为它需要与当前用户的 Wayland 会话交互。
- USB 首版不实现自定义 USB Bulk 协议。MVP 采用 `adb reverse` 或等价 ADB 端口转发方案，降低 Android 端与 Linux 端实现复杂度。
- 编码格式首版强制要求 `H.264` 可用。`H.265/HEVC` 为可选增强。`AV1` 延后。
- 首版默认只支持 SDR 8-bit 串流链路。配置中可保留色深字段，但仅当后端与编解码器都支持时才可启用更高位深。
- 首版一台服务端只维护一个活动副屏会话。
- 虚拟显示器必须懒创建: 未连接客户端时不创建显示器；客户端断开后必须销毁对应显示器。
- 服务端必须为每个客户端生成名称稳定的虚拟显示器标识，以便 Linux 桌面环境可记住该客户端对应的布局、缩放和位置配置。

**范围定义**

MVP 必须交付以下能力。

- Rust CLI 服务端。
- Flutter Android 客户端。
- Wayland 环境下可用的单副屏工作流。
- 按客户端连接生命周期动态创建和销毁虚拟显示器。
- 面向每个客户端的稳定命名虚拟显示器。
- 基于配置文件的服务端行为控制。
- `systemd --user` unit 文件与运行说明。
- USB 和局域网两种连接方式。
- 客户端设置页与服务端参数协商。
- 客户端上报设备屏幕参数，并作为虚拟屏幕默认参数来源。
- 客户端记住上次成功连接的服务端地址，并在打开应用时自动尝试连接。
- 基础诊断命令、日志与错误提示。

Post-MVP 可扩展能力。

- 多显示器配置与多配置档案。
- 自动发现服务端 `mDNS`。
- UDP/QUIC 低延迟媒体通道。
- 触控映射与输入回传。
- 音频、剪贴板、文件拖放。
- 更多 Wayland compositor 适配。

**术语定义**

- 虚拟显示器: Linux 桌面环境中额外暴露出来的一块可扩展输出。
- 显示参数: 虚拟显示器自身参数，例如分辨率、刷新率、逻辑尺寸、方向、色深。
- 设备屏幕参数: 客户端实际上报的本机屏幕能力，例如物理分辨率、刷新率、色深、方向和 DPI。
- 串流参数: 视频输出链路参数，例如编码分辨率、帧率、编码器、码率、GOP、B 帧、低延迟模式。
- 服务器偏好: 服务端配置中声明的默认/推荐串流参数。
- 客户端覆盖: 客户端在单次会话中申请替代服务器偏好的串流请求。
- 最终生效参数: 服务端结合硬件能力、配置上限和客户端请求后确认的实际运行参数。

**系统概览**

系统由 5 个核心模块构成。

- 显示后端模块: 负责创建、接入或管理虚拟显示输出。
- 捕获模块: 负责从目标输出获取原始视频帧。
- 编码与串流模块: 负责格式转换、编码、码率控制、分片和发送。
- 控制协议模块: 负责握手、鉴权、能力交换、参数协商、会话状态同步。
- Android 客户端模块: 负责连接、解码、全屏渲染、设置 UI 和重连逻辑。

推荐的逻辑数据流如下。

1. 服务端启动并加载配置。
2. 客户端通过 USB 或 LAN 发起握手，并上报自身设备屏幕参数与解码能力。
3. 服务端识别客户端身份，计算该客户端对应的稳定虚拟显示器名称。
4. 服务端按客户端参数或服务端强制覆盖配置创建虚拟显示器。
5. Linux 桌面环境将该虚拟显示器视为一块新接入的显示器，应用已记忆的布局配置或等待用户配置。
6. 双方交换能力与偏好，完成串流参数协商。
7. 服务端返回最终生效参数并开始串流。
8. 客户端解码并全屏显示。
9. 客户端调整设置后，发送重新协商请求。
10. 服务端平滑重建或更新编码会话。
11. 客户端断开后，服务端销毁对应虚拟显示器。

**平台与兼容性要求**

- 服务端平台: Linux x86_64 优先，Wayland 桌面会话。
- Android 客户端平台: Android 10 及以上优先支持，Android 8/9 可作为后续兼容目标。
- Wayland compositor: v1 需要明确列出支持矩阵，建议优先支持一类 compositor，再通过后端抽象扩展。
- 服务端运行账户: 默认当前桌面登录用户。

**Wayland 支持策略**

这是产品最关键的技术风险点，PRD 做出以下限制。

- v1 不要求“任何 Wayland 环境都可用”。
- v1 必须实现 `DisplayBackend` 抽象层，避免后续被单一后端锁死。
- v1 推荐优先支持一种可验证的后端路径，例如基于 `VKMS/DRM` 的虚拟输出方案，或单一 compositor 的专用虚拟输出能力。
- 如果运行环境不支持当前后端，服务端必须在启动时给出明确报错，而不是在串流阶段静默失败。
- CLI 必须提供 `doctor` 或 `probe` 诊断命令，输出当前会话环境、可用后端、编码器能力和缺失依赖。
- 所选后端必须支持“按需创建/销毁虚拟显示器”和“为同一客户端重复创建名称稳定的显示器标识”这两个行为，否则不能作为 v1 主后端。

**服务端产品需求**

服务端必须是一个 CLI 工具，推荐命令结构如下。

- `tab-screen serve`: 启动主服务。
- `tab-screen doctor`: 诊断 Wayland、编码器、ADB、网络与权限状态。
- `tab-screen probe`: 输出可用显示后端、编码器、分辨率与设备能力。
- `tab-screen print-default-config`: 输出默认配置模板。
- `tab-screen usb adb-reverse`: 建立 USB ADB 端口转发。
- `tab-screen version`: 输出版本与构建信息。

服务端必须支持配置文件。

- 优先级: CLI 参数 > 环境变量 > 配置文件 > 内建默认值。
- 默认用户级路径: `~/.config/tab-screen/config.toml`
- 默认系统级路径: `/etc/tab-screen/config.toml`
- 服务端启动时必须打印实际加载的配置路径。
- 配置中必须允许用户声明“跟随客户端屏幕参数”或“强制覆盖显示参数”。

服务端必须支持以下配置域。

- 基础服务配置: 监听地址、端口、日志等级、会话超时。
- 显示后端配置: 后端类型、虚拟显示启用方式、按连接创建策略、客户端身份映射策略、稳定命名策略。
- 虚拟显示参数: 是否跟随客户端设备屏幕参数、分辨率列表、默认分辨率、刷新率列表、默认刷新率、色深、旋转、缩放比例、DPI、强制覆盖规则。
- 串流偏好参数: 默认分辨率、默认帧率、默认码率、默认编码格式、低延迟模式。
- 串流限制参数: 最大分辨率、最大帧率、允许编码格式列表、码率范围、是否允许客户端覆盖。
- 编码器配置: `hardware/software/auto`、设备选择、GOP 长度、关键帧间隔、B 帧开关、preset、profile、tune。
- 连接与安全配置: 配对模式、认证 token、是否要求加密、允许的客户端列表。
- USB 配置: ADB 可执行路径、默认转发端口、自动建立 reverse 开关。
- 运行与恢复配置: 自动重连、崩溃恢复、客户端身份映射持久化与最近连接记忆行为。

服务端必须具备以下功能行为。

- 启动时校验配置合法性，并在错误时给出精确字段和原因。
- 能够在没有客户端连接时保持待机状态。
- 在客户端连接前不得创建虚拟显示器。
- 在客户端连接并完成最小握手后，才创建该客户端对应的虚拟显示器和串流会话。
- 必须依据客户端上报的设备屏幕参数生成虚拟显示默认值，除非服务端配置显式强制覆盖相关参数。
- 必须为每个客户端创建名称稳定的虚拟显示器，确保同一客户端多次连接时 Linux 桌面能识别为同一逻辑显示设备。
- 在客户端断开后释放编码资源，并立即销毁对应虚拟显示器。
- 在收到客户端覆盖请求时执行参数协商。
- 在协商成功时返回最终生效参数。
- 在协商失败时返回拒绝原因和可用替代选项。
- 支持在会话中重新协商部分参数，例如帧率、码率、编码器和分辨率。
- 必须向客户端暴露当前实际状态，例如连接中、已连接、串流中、重协商中、错误。
- 必须持久化每个客户端的稳定身份与虚拟显示名称映射，避免重启服务后名称漂移。

**客户端产品需求**

Android 客户端必须以 Flutter 实现，并在视觉与交互上符合现代 Android 应用风格。

客户端必须包含以下页面。

- 首页: 展示连接状态、USB 连接入口、局域网连接入口、最近连接记录。
- 连接页: 输入服务端 `IP:port`，执行连接、鉴权、错误提示。
- 全屏显示页: 以沉浸式模式显示副屏画面，支持保持屏幕常亮。
- 设置页: 配置“遵循服务器偏好”或手动覆盖串流参数。
- 诊断页: 查看当前连接方式、实际生效参数、延迟、帧率、丢帧、解码方式、错误日志摘要。

客户端必须支持以下设置项。

- 连接模式: USB / LAN。
- 服务器地址和端口。
- 是否遵循服务器偏好。
- 请求串流分辨率。
- 请求帧率。
- 请求编码格式。
- 请求色深。
- 请求码率上限或质量档位。
- 是否优先硬件解码。
- 是否在启动后自动进入全屏。
- 是否显示调试 overlay。

客户端必须具备以下行为。

- 启动时必须上报本机设备屏幕参数与解码能力。
- 连接成功后立即显示协商后的实际参数。
- 当用户修改设置时向服务端发送重新协商请求。
- 当服务端拒绝请求时在 UI 中显示原因，并保持当前会话或提示重新连接。
- 支持自动重连和用户手动重连。
- 必须记住最近一次成功连接的 `IP:port`，并在用户打开应用时自动尝试连接该地址。
- 支持在 Android 设备旋转时保持画面稳定，不出现错误拉伸。
- 在解码失败、网络断开、认证失败时显示明确错误信息。
- 使用平台硬件解码能力时优先启用 `MediaCodec`，失败时允许回退软件解码或提示不支持。

**连接方式需求**

USB 连接 v1 采用 ADB 端口转发方案。

- 前提条件: Android 开启开发者选项和 USB 调试。
- 服务端 CLI 可帮助用户自动执行 `adb reverse tcp:<port> tcp:<port>`。
- 客户端在 USB 模式下默认连接 `127.0.0.1:<port>`。
- 若 ADB 未安装、设备未授权或 reverse 失败，服务端必须给出明确诊断结果。

局域网连接要求如下。

- 客户端可手动输入 `IP:port`。
- 服务端默认绑定本机可配置地址。
- 服务端可选开启仅本地子网访问限制。
- 局域网模式必须具备认证能力，避免同网段其他设备无授权接入。

**控制协议需求**

控制协议必须满足以下要求。

- 有明确版本号，允许未来升级。
- 具备握手、能力交换、会话创建、重协商、心跳和错误上报消息。
- 必须包含客户端稳定身份与客户端设备屏幕参数上报字段。
- 能够表达客户端请求值、服务端限制值和最终生效值。
- 能够表达失败原因代码与人类可读消息。
- 同一协议应同时适用于 USB 与 LAN。

推荐控制消息模型如下。

- `ClientHello`: 客户端版本、平台、客户端稳定身份、设备型号、支持的编解码器、解码能力、网络模式、设备屏幕参数。
- `ServerHello`: 服务端版本、可用显示后端、可用编码器、服务器偏好、限制范围、该客户端对应的稳定虚拟显示器名称。
- `StartSessionRequest`: 客户端请求的显示目标与串流参数。
- `StartSessionResponse`: 接受或拒绝，若接受则返回最终生效参数。
- `UpdateStreamRequest`: 会话中重协商请求。
- `UpdateStreamResponse`: 返回生效结果或拒绝原因。
- `Heartbeat`: 保活和状态统计。
- `Error`: 错误码、消息、是否可恢复。
- `SessionEnded`: 服务端或客户端主动结束会话。

推荐错误码类别如下。

- 认证错误。
- 协议版本不兼容。
- 配置非法。
- 显示后端不可用。
- 编码器不可用。
- 请求参数超出限制。
- 解码器不可用。
- 网络中断。
- 会话超时。

**媒体链路需求**

媒体链路必须满足以下要求。

- 至少支持 `H.264` 编码。
- 必须支持服务端自动选择硬件编码、软件编码或显式指定模式。
- 必须支持基础码率控制参数。
- 必须支持关键帧策略配置。
- 客户端必须能显示当前实际使用的编码格式、分辨率、帧率和解码方式。
- 必须将虚拟显示器参数决策和串流参数决策分为两个阶段，避免两者互相混淆。

媒体参数要求如下。

- 虚拟显示器参数与串流参数必须解耦。
- 在默认模式下，虚拟显示器的分辨率、刷新率和色深应来自客户端上报的设备屏幕参数。
- 服务端可通过配置对虚拟显示器参数执行显式强制覆盖；若配置未覆盖，则不得忽略客户端上报值而任意选择默认显示参数。
- 允许虚拟显示器为 `2560x1600@60`，串流实际输出为 `1920x1200@30`。
- 当服务端资源不足或客户端能力不够时，可在保持虚拟显示器不变的情况下下调串流参数。
- 如果参数变化需要重建编码器，服务端应尽量平滑切换，并通知客户端短暂黑屏或重连原因。

编码支持策略如下。

- v1 必选: `H.264`, 8-bit, SDR。
- v1 可选: `H.265/HEVC`, 8-bit, SDR。
- Post-MVP: `AV1`, HDR, 10-bit。

**参数协商规则**

参数协商必须遵循以下逻辑。

1. 服务端读取自身配置与本机能力，形成“服务器能力集”。
2. 客户端上报稳定身份、设备屏幕参数、解码能力与用户请求，形成“客户端请求集”。
3. 服务端先决策虚拟显示器参数: 默认跟随客户端设备屏幕参数，若存在服务端强制覆盖则以强制覆盖为准。
4. 服务端创建该客户端对应的稳定命名虚拟显示器。
5. 服务端再根据硬限制过滤非法串流参数。
6. 若客户端选择“遵循服务器偏好”，则优先采用服务器偏好值。
7. 若客户端提供覆盖值，服务端在允许范围内尽量满足。
8. 服务端返回最终生效值和降级原因。

降级示例。

- 客户端请求 `4K@60 HEVC`，服务端只允许 `1080p@60 H.264`，则返回降级后的实际值与原因。
- 客户端请求 10-bit，服务端编码器仅支持 8-bit，则降级为 8-bit 并给出说明。

**配置模型要求**

服务端配置文件格式推荐为 `TOML`。

必须支持如下概念。

- 默认值。
- 允许范围。
- 硬性禁止项。
- 自动模式与显式模式。
- 面向未来扩展的版本化配置。

配置示例如下。

```toml
[server]
listen_host = "0.0.0.0"
listen_port = 38491
log_level = "info"
session_timeout_secs = 15

[display]
backend = "auto"
name_prefix = "Tab Screen"
create_on_client_connect = true
destroy_on_client_disconnect = true
identity_source = "client_stable_id"
remember_per_client_display = true
follow_client_screen_params = true
force_resolution = ""
force_refresh_rate = 0
force_color_depth = 0
supported_resolutions = ["1920x1200", "2560x1600"]
supported_refresh_rates = [60]
allowed_color_depths = [8]
dpi_mode = "follow-client"

[stream.preference]
resolution = "1920x1200"
frame_rate = 60
codec = "h264"
bitrate_kbps = 12000
encoder = "auto"
low_latency = true

[stream.limits]
allow_client_override = true
max_resolution = "2560x1600"
max_frame_rate = 60
allowed_codecs = ["h264", "hevc"]
allowed_color_depths = [8]
min_bitrate_kbps = 2000
max_bitrate_kbps = 30000

[encoder]
mode = "auto"
device = ""
gop = 60
b_frames = 0
preset = "low-latency"
profile = "high"
tune = "zerolatency"

[network]
require_auth = true
token = "change-me"
allow_lan = true
allow_usb = true

[usb]
adb_path = "adb"

auto_reverse = true
```

**Systemd 需求**

服务端必须提供 `systemd --user` 集成方案。

- 必须提供示例 unit 文件。
- unit 需支持开机后在用户图形会话可用时自动启动。
- unit 必须能与 `journalctl --user` 配合查看日志。
- 文档中必须说明为何推荐用户级 service，而不是系统级 service。
- unit 启动后的默认行为必须是“待机等待客户端连接”，而不是预先创建虚拟显示器。

推荐 unit 示例。

```ini
[Unit]
Description=Tab Screen Server
After=graphical-session.target
PartOf=graphical-session.target

[Service]
ExecStart=%h/.local/bin/tab-screen serve --config %h/.config/tab-screen/config.toml
Restart=on-failure
RestartSec=2
Environment=RUST_LOG=info

[Install]
WantedBy=default.target
```

**用户体验要求**

客户端 UI 需满足以下视觉与交互要求。

- 采用现代、简洁、偏工具型设计风格。
- 默认支持深色模式，并兼容浅色模式。
- 全屏显示页默认隐藏系统 UI，保留可唤出的最小控制层。
- 关键状态必须可见，例如“已连接”“正在重连”“参数已降级”。
- 设置页必须清楚区分“服务器偏好”和“客户端覆盖值”。
- 不允许让用户误以为自己设置的参数一定已经生效，必须显示“请求值”和“实际值”的区别。
- 首页应突出展示“上次成功连接的服务端”并在应用打开时自动尝试重连。

**可观测性与诊断要求**

服务端必须提供结构化日志，至少覆盖以下事件。

- 启动与配置加载。
- 显示后端初始化。
- 客户端身份识别与稳定显示器命名决策。
- 虚拟显示器创建与销毁。
- 编码器选择与失败回退。
- ADB 操作结果。
- 客户端连接与断开。
- 会话协商与重协商。
- 致命错误与可恢复错误。

客户端必须提供基础运行指标显示。

- 当前连接方式。
- 当前编码格式。
- 当前分辨率。
- 当前帧率。
- 估算延迟。
- 解码方式。
- 最近错误。

**性能要求**

MVP 目标性能如下。

- 在受支持硬件上，`1920x1200 H.264 30fps` 应可稳定运行 30 分钟无崩溃。
- 在具备硬件编码和硬件解码的环境下，目标支持 `1920x1200 60fps`。
- 端到端交互延迟目标: USB 模式优于 100ms，局域网模式优于 150ms。
- 服务端在空闲无客户端时 CPU 占用应接近空闲状态。
- 客户端在全屏显示期间不应频繁触发界面重建导致卡顿。

这些指标为目标值，不作为所有设备上的绝对承诺。实现阶段必须记录测试设备和结果。

**安全要求**

- 局域网模式默认必须启用认证。
- 禁止匿名客户端直接获取桌面画面。
- token 或配对凭据必须支持配置化和持久化。
- 服务端日志不得泄露敏感画面内容。
- 若未来增加加密传输，协议层必须保留扩展位。

**失败处理要求**

系统必须对以下失败情形提供可预期行为。

- Wayland 会话不可用: 启动失败并提示当前环境不满足条件。
- 虚拟显示后端不可用: 启动失败并给出诊断建议。
- 客户端未上报必要的设备屏幕参数: 拒绝创建虚拟显示器并返回协议错误。
- 客户端身份不稳定或映射持久化失败: 明确告警，避免显示器命名漂移导致桌面布局丢失。
- 编码器不可用: 尝试回退软件编码，若失败则明确退出。
- ADB 不可用: USB 模式标记不可用，但不影响 LAN 模式。
- 网络断开: 客户端提示断开并进入重连流程。
- 客户端请求超限: 服务端拒绝并返回可选范围。
- Android 解码器不支持: 客户端提示不支持并建议改用 `H.264` 或更低参数。

**版本范围与里程碑**

建议按以下阶段实施。

阶段 1: 基础能力验证。

- 明确一个可行的 Wayland 虚拟显示后端。
- 验证该后端支持按连接生命周期创建/销毁显示器，并支持稳定命名。
- 打通从虚拟显示到编码输出的最小链路。
- 定义并实现基础控制协议。
- Android 端完成最小可用播放器和连接页。

阶段 2: MVP。

- 完成配置文件、CLI、systemd、ADB reverse、LAN 连接。
- 完成客户端屏幕参数上报、参数协商、设置页、自动重连、日志、诊断命令。
- 完成 `H.264` 软硬编码路径。

阶段 3: 增强。

- 增加 `HEVC`。
- 增强 compositor 兼容性。
- 增加更细粒度性能统计。

**验收标准**

MVP 验收必须满足以下条件。

- 可以通过 `tab-screen serve` 在受支持 Wayland 环境中启动服务。
- 可以通过 `systemd --user` 管理服务的启动、停止、重启和日志查看。
- 服务端可以从配置文件中读取虚拟显示参数和串流参数。
- 在默认配置下，服务端会使用客户端上报的分辨率、刷新率和色深创建虚拟显示器；在强制覆盖配置下，则使用服务端配置值。
- Android 客户端可以通过 LAN 手动输入 `IP:port` 成功连接。
- Android 客户端可以通过 USB 模式在 ADB reverse 就绪后成功连接。
- Android 客户端会记住上次成功连接的 `IP:port`，并在打开应用时自动尝试连接。
- 客户端连接成功后 Linux 桌面会自动出现一个新的扩展显示器，行为接近插入实体显示器。
- 客户端断开后，对应虚拟显示器会被销毁。
- 同一客户端多次连接时，服务端创建的虚拟显示器名称保持稳定，Linux 桌面可以记住该客户端的显示布局和缩放配置。
- 客户端可以在设置页选择“遵循服务器偏好”或手动覆盖参数。
- 服务端会将最终生效参数返回客户端并显示在 UI 中。
- 会话中调整部分串流参数后，服务端能够完成重协商或明确拒绝。
- 在至少一套受支持设备组合上，`1080p30 H.264` 稳定运行 30 分钟。
- 在连接失败、认证失败、编码失败、后端不可用时，用户能看到明确错误。

**建议的仓库与模块拆分**

以下不是硬性产品要求，但建议后续实现遵循。

- `crates/protocol`: 协议数据结构、版本定义、错误码。
- `crates/server-core`: 配置、会话管理、协商逻辑、日志。
- `crates/display-backend`: 虚拟显示抽象与具体后端实现。
- `crates/encoder`: 编码器抽象、硬件/软件实现。
- `crates/transport`: 控制通道与媒体通道封装。
- `apps/android`: Flutter 客户端。
- `apps/android/native`: Android 平台解码与 Texture 桥接插件。

**推荐的实现优先级**

为了降低风险，后续 Agent 应优先完成以下顺序。

1. 先验证 Wayland 虚拟显示后端，而不是先做 UI。
2. 先完成固定参数串流闭环，再引入参数协商。
3. 先完成 LAN 模式，再复用同一协议接入 USB ADB reverse。
4. 先完成 `H.264`，再扩展 `HEVC`。
5. 先完成 `systemd --user`，再考虑系统级部署变体。

**开放风险说明**

以下内容不应阻塞 PRD 落地，但实现前必须先验证。

- 选定的 Wayland 虚拟显示方案在目标 compositor 上是否稳定。
- 所选 Rust 编码栈与 Flutter Android 解码链路之间的封装复杂度。
- USB ADB reverse 在不同 Android ROM 和 Linux 发行版上的稳定性。
- Android 硬件解码器对高分辨率和高帧率的实际能力差异。

**最终产品定义**

`tab-screen` v1 是一个面向 Linux Wayland 的本地副屏系统。

- 服务端是一个由 Rust 编写、可由 `systemd --user` 管理、通过 `TOML` 配置文件控制行为的 CLI 工具。
- 客户端是一个由 Flutter 编写、面向 Android 平板、具有现代化 UI 的专用副屏应用。
- v1 明确支持单副屏、USB ADB reverse 与 LAN 连接、`H.264` 串流、参数协商与诊断。
- v1 明确要求客户端上报自身屏幕参数、服务端按连接创建和销毁虚拟显示器，并为每个客户端提供稳定命名的显示设备。
- v1 明确不承诺通用 Wayland 全兼容、多副屏、远程输入和音频等高级能力。

后续 Agent 在编码时应严格以“先验证显示后端，再形成最小串流闭环，再补齐配置、协商和 UI”的顺序推进。
