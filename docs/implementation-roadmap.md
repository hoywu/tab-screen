# Tab Screen Implementation Roadmap

**文档信息**

- 项目代号: `tab-screen`
- 文档类型: Implementation Roadmap
- 文档目标: 将 `docs/prd.md` 和 `docs/architecture.md` 拆解为可执行的分阶段编码任务
- 适用范围: MVP 到首个可验收版本
- 当前版本: `v1.1`
- 文档状态: Ready for implementation

**执行原则**

- 永远先验证显示后端，再扩展 UI。
- 永远先打通固定参数串流闭环，再加入协商和重配置。
- 永远先打通 LAN，再复用到 USB `adb reverse`。
- 永远先让 `H.264` 成为稳定默认路径，再考虑 `HEVC`。
- 永远在编码同一改动中补齐必要测试或验证记录，不把测试留到阶段收尾。
- 每个阶段结束都必须有可验证产物，未过门槛不进入下一阶段。

**里程碑总览**

| 阶段 | 名称 | 目标 | 退出门槛 |
| --- | --- | --- | --- |
| Phase 0 | 仓库骨架与基线 | 建立目录、接口、配置和协议骨架 | 工程可编译，骨架命令可运行 |
| Phase 1 | 显示后端与捕获验证 | 证明主后端可创建/销毁稳定命名显示器并可捕获 | 生成验证记录，后端路径被确认 |
| Phase 2 | 最小闭环 | 固定参数 LAN 串流跑通 | Android 可看到 Linux 副屏画面 |
| Phase 3 | 协商与稳定命名 | 完成身份持久化、参数协商、重协商 | 同一客户端可稳定重连并得到实际参数 |
| Phase 4 | 产品化客户端 | 完成设置页、自动重连、诊断页、全屏体验 | 客户端满足 MVP UI 要求 |
| Phase 5 | 运维与接入 | 完成系统级服务集成、doctor/probe、USB ADB reverse | 服务端具备部署和诊断能力 |
| Phase 6 | MVP 验收与硬化 | 完成长稳验证、错误处理、文档补齐 | 对照 PRD 验收项通过 |

**工作流约束**

后续 Agent 按以下顺序推进，不要跳步。

1. 先完成 `crates/` 和 `apps/android/` 的骨架，再开始具体实现。
2. 先确定主显示后端，再开始会话管理和 UI 细化。
3. 先实现“固定参数 + 手动连接 + 单次会话”，再加自动重连和设置。
4. 先让链路可观测，再补性能优化。
5. 每完成一个完整模块后，必须同步更新 `docs/implementation-status.md`，作为后续 Agent 的交接入口。
6. 每完成一个完整模块后，必须补齐与该模块相匹配的必要测试；若暂时无法自动化，必须留下明确的人工验证记录。

**Phase 0: 仓库骨架与基线**

目标是把“可以开始写功能”的基础地基搭起来。

交付物。

- Rust workspace 与各 crate 骨架。
- Flutter Android App 骨架和页面占位。
- 统一协议模型 crate。
- 统一配置模型 crate。
- CLI 命令骨架。
- 日志初始化和错误码枚举。

服务端任务。

- 创建 Cargo workspace。
- 创建 `protocol/config/server-core/display-backend/capture/encoder/transport/app-cli` crate。
- 在 `protocol` 中定义消息枚举、协商模型、错误码和基础类型。
- 在 `config` 中定义 TOML 模型、默认值和校验接口。
- 在 `app-cli` 中建立命令入口: `serve/doctor/probe/print-default-config/usb/version`。
- 引入 `tracing` 和统一错误类型。

客户端任务。

- 创建 Flutter App。
- 建立页面路由和基础主题。
- 建立 `SessionController`、`SettingsController`、`DiagnosticsController` 占位。
- 建立 `PreferencesRepository`，先只支持本地稳定 ID 和上次地址存储。
- 搭建 Android 原生插件工程壳。

验收标准。

- `cargo check` 通过。
- Flutter App 可在 Android 模拟器或设备启动。
- CLI 命令能输出帮助或占位结果。
- 协议和配置模型可被两端共享或映射。

**Phase 1: 显示后端与捕获验证**

这是最高风险阶段，必须最先做。

目标。

- 确认一个可用的主 `DisplayBackend`。
- 证明其能按连接生命周期创建和销毁虚拟显示器。
- 证明同一客户端可复用稳定命名显示器。
- 证明该显示器可被可靠捕获。

必须产出的验证结论。

- 支持的 compositor/环境矩阵。
- 主后端的依赖条件和失败方式。
- 是否需要额外权限、模块、内核能力或系统服务前置条件。
- 稳定命名是原生支持还是通过映射层实现。

具体任务。

- 在 `display-backend` 中定义 `DisplayBackend` 与 `DisplayHandle`。
- 实现至少一个候选后端的 `probe`。
- 编写独立验证命令或测试程序，执行“创建显示器 -> 等待 -> 销毁显示器”。
- 为固定逻辑显示名执行多次创建销毁，确认名字不漂移。
- 在 `capture` 中实现最小帧获取路径。
- 输出一份验证记录文档，写入 `docs/` 或测试报告文件。

退出门槛。

- 能够在目标环境中重复完成创建/销毁。
- 对同一稳定 ID 能得到同一逻辑显示名。
- 能捕获到非空视频帧。
- 已明确“支持环境”和“启动前置条件”。

如果失败。

- 立即回退并更换后端路线。
- 不进入 Phase 2。

**Phase 2: 最小闭环**

目标是打通固定参数 LAN MVP 路径，不做完整协商。

范围。

- 单会话。
- 手动输入 `IP:port`。
- 固定 `H.264` 参数。
- 单条 WebSocket 连接。
- 客户端全屏显示。

服务端任务。

- 在 `transport` 中实现 WebSocket 监听和文本/二进制帧分流。
- 在 `server-core` 中实现最小状态机: `Idle -> Handshaking -> Streaming -> Terminating`。
- 暂时用固定参数创建显示器和编码器配置。
- 将捕获帧编码为 H.264 Annex B 并推送给客户端。
- 实现基础心跳和断连清理。

客户端任务。

- 连接页支持手动输入地址。
- 能建立 WebSocket 连接并发送最小 `client_hello`。
- 接收服务端固定参数响应。
- Android 原生插件实现最小 `MediaCodec` H.264 解码。
- 全屏页显示解码后画面。

验证点。

- 客户端连接后，Linux 桌面出现新显示器。
- 客户端能稳定看到该显示器画面。
- 断开连接后显示器被销毁。

退出门槛。

- LAN 模式闭环在至少一套环境跑通。
- 单次会话无明显资源泄露。
- 客户端黑屏恢复、断连提示和服务端清理至少具备基础行为。

**Phase 3: 协商与稳定命名**

目标是把 PRD 中最核心的“像一块真实显示器一样接入”的行为做完整。

服务端任务。

- 完成配置加载优先级实现。
- 完成 `clients.toml` 身份映射持久化。
- 实现显示参数第一阶段决策。
- 实现串流参数第二阶段协商。
- 实现 `start_session_request/response`。
- 实现 `update_stream_request/response`。
- 实现降级原因生成和错误码回传。
- 实现 `session_busy` 和单会话拒绝逻辑。

客户端任务。

- 上报完整设备屏幕参数和解码能力。
- 设置页支持“遵循服务器偏好”和手动覆盖值。
- 展示“请求值”和“实际值”。
- 在设置变更后触发 `update_stream_request`。
- 处理服务端拒绝并保留当前会话。

测试任务。

- 协商单元测试覆盖服务端偏好、客户端覆盖、超限降级、拒绝路径。
- 稳定命名集成测试覆盖“重启服务后名称仍稳定”。

退出门槛。

- 同一客户端多次连接显示名稳定。
- 默认模式下虚拟显示参数跟随客户端屏幕参数。
- 强制覆盖配置可生效。
- 客户端能看到服务端最终生效参数。
- 会话中至少一种参数可完成重协商。

**Phase 4: 产品化客户端**

目标是让 Android 端从“验证工具”变成“可日常使用的 MVP 客户端”。

任务。

- 首页展示连接状态、最近连接记录、上次成功地址。
- 应用启动时自动尝试连接上次成功地址。
- 支持 USB/LAN 连接模式切换。
- 完成设置页视觉整理，明确区分服务器偏好和客户端覆盖。
- 完成诊断页，显示连接方式、参数、解码方式、延迟估算、最近错误。
- 完成沉浸式全屏模式和屏幕常亮。
- 完成重连中的 UI 状态和错误提示文案。

质量要求。

- 页面状态切换不应导致频繁重建播放器。
- 旋转屏幕时画面保持稳定，不出现错误拉伸。
- 自动重连不应造成重复会话创建。

退出门槛。

- 客户端具备 PRD 要求的 5 个页面。
- 最近成功地址可自动重连。
- 设置页和诊断页信息完整可读。

**Phase 5: 运维与接入**

目标是补齐服务端真正可部署、可诊断、可通过 USB 使用的能力。

任务。

- 实现 `doctor`，输出环境检查和修复建议。
- 实现 `probe`，输出后端和编码能力。
- 实现 `print-default-config`。
- 实现 `usb adb-reverse` 命令。
- 为 USB 模式复用同一 WebSocket 会话协议。
- 提供系统级服务 unit 示例和说明文档。
- 明确系统级特权服务所需的权限、状态目录和日志查看方式。
- 确保 `serve` 作为待机服务运行时不会预创建显示器。

测试任务。

- `doctor` 在缺失显示后端、缺失 ADB、后端不可用、权限不足等场景下给出明确输出。
- `adb reverse` 成功和失败路径都有清晰日志与返回码。
- 系统级服务启动后能在待机状态下保持可诊断且不预创建显示器。

退出门槛。

- 用户可用 `tab-screen serve` 启动服务。
- 用户可用系统级服务管理器管理服务。
- USB 模式在 `adb reverse` 就绪后可以成功连接。
- `doctor/probe` 能帮助定位主要失败场景。

**Phase 6: MVP 验收与硬化**

目标是对照 PRD 完成真正的 MVP 收口，而不是继续加新功能。

任务。

- 对照 `docs/prd.md` 验收标准逐项建立检查清单。
- 执行至少一套支持设备组合上的 30 分钟稳定性测试。
- 验证 `1080p30 H.264` 稳定运行。
- 验证 LAN 和 USB 两种连接方式。
- 验证错误路径: 认证失败、后端不可用、编码失败、参数超限、解码失败、网络断开。
- 清理日志和错误文案，确保诊断信息可读。
- 补齐使用文档和运行说明。

退出门槛。

- 满足 PRD 中 MVP 验收项。
- 所有关键错误场景能看到明确提示。
- 长稳测试结果已记录。

**建议的任务拆分顺序**

为了减少冲突，后续 Agent 可按下面顺序领取任务。

1. Agent A: Rust workspace、协议、配置、CLI 骨架。
2. Agent B: Flutter App 骨架、页面路由、存储层。
3. Agent C: 显示后端 probe 和创建/销毁验证。
4. Agent D: 捕获 + 编码最小链路。
5. Agent E: WebSocket 传输与最小状态机。
6. Agent F: Android 原生解码插件。
7. Agent G: 参数协商、稳定命名和持久化。
8. Agent H: 设置页、诊断页和自动重连。
9. Agent I: doctor/probe/系统级服务/USB。
10. Agent J: 集成测试、验收清单和文档补齐。

其中 3/4/5/6 之间存在较强依赖，必须围绕同一条最小闭环频繁合并验证，不建议各自长期分叉。

**每阶段必须保留的产物**

- 可运行代码。
- 与改动匹配的测试代码，或因外部限制暂无法自动化时的验证记录。
- 验证命令或测试结果。
- 关键设计决策记录。
- 已知限制和下一阶段风险。

如果某阶段只有代码、没有测试或验证记录，视为阶段未完成。

**明确延期到 Post-MVP 的内容**

- `HEVC` 正式支持。
- 多 compositor 扩展。
- QUIC/UDP 媒体通道。
- 多客户端/多副屏。
- 音频、输入、剪贴板、文件传输。
- HDR、10-bit、AV1。

**最终执行口径**

后续 Agent 按本路线图推进时，应遵守两条硬规则。

1. 任何时候只要主显示后端可行性存疑，就暂停新增功能，优先补验证。
2. 任何时候只要最小闭环未稳定，就不要扩展 HEVC、发现服务端、复杂 UI 动效等增强项。
3. 任何功能实现或缺陷修复，只要缺少必要测试或明确验证结果，就不要标记为完成。
